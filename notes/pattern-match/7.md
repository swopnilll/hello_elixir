# Pipe Operator (`|>`) in Elixir (Notes)

---

## 🧠 Goal

Refactor repetitive function calls into **clean, readable flow**

---

## 🔹 Before (Without Pipe)

```elixir id="r1"
def create_hand(hand_size) do
  deck = create_deck()
  deck = shuffle(deck)
  {hand, _rest} = deal(deck, hand_size)
  hand
end
```

### ❌ Problems

* Repetition (`deck` everywhere)
* Harder to read
* Not idiomatic Elixir

---

## 🔥 After (Using Pipe Operator)

```elixir id="r2"
def create_hand(hand_size) do
  create_deck()
  |> shuffle()
  |> deal(hand_size)
end
```

---

## 🔍 How Pipe Works

```elixir id="r3"
value |> function()
```

👉 Equivalent to:

```elixir id="r4"
function(value)
```

---

## 🔁 Step-by-Step Flow

```elixir id="r5"
create_deck()
|> shuffle()
|> deal(hand_size)
```

### Execution:

1️⃣

```elixir id="r6"
create_deck()
```

→ returns deck

2️⃣

```elixir id="r7"
shuffle(deck)
```

3️⃣

```elixir id="r8"
deal(deck, hand_size)
```

---

## 🧠 Key Rule

> Pipe passes result as **FIRST argument**

---

## 🔹 Important Detail

```elixir id="r9"
|> deal(hand_size)
```

Elixir internally does:

```elixir id="r10"
deal(previous_result, hand_size)
```

---

## ⚠️ Critical Requirement

Functions MUST accept:

> Data as the **first argument**

---

## ❌ Bad Design (Breaks Pipe)

```elixir id="r11"
def deal(hand_size, deck) do
end
```

* Pipe would fail ❌
* Because deck is not first argument

---

## ✅ Good Design (Pipe-Friendly)

```elixir id="r12"
def deal(deck, hand_size) do
end
```

* Works perfectly with pipe ✅

---

## 🔹 Output of Function

```elixir id="r13"
create_hand(5)
```

Returns:

```elixir id="r14"
{hand, remaining_deck}
```

---

## 🧠 Mental Model

```text id="r15"
Data → Function → Function → Function
```

Instead of:

```text id="r16"
Function(data) → Function(result)
```

---

## 🔥 Why Pipe Operator is Powerful

* Removes temporary variables
* Improves readability
* Encourages functional style
* Makes data flow obvious

---

## 🆚 JavaScript Comparison

### JavaScript (Nested / Imperative)

```javascript id="r17"
const deck = createDeck();
const shuffled = shuffle(deck);
const result = deal(shuffled, handSize);
```

---

### Elixir (Pipe)

```elixir id="r18"
create_deck()
|> shuffle()
|> deal(hand_size)
```

---

## 🧠 Big Insight

> Pipe enforces better function design

* Consistent argument order
* Cleaner APIs
* Predictable flow

---

## 🔹 Best Practice

* Always design functions like:

```elixir id="r19"
function(data, other_args)
```

NOT:

```elixir id="r20"
function(other_args, data)
```

---

## 🧠 Takeaways

* `|>` passes result forward
* Always goes into **first argument**
* Reduces repetition
* Improves readability
* Core concept in Elixir

---

## 🚀 Final Thought

> Pattern Matching + Pipe Operator = Core of Elixir
