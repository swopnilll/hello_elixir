# Atoms + Advanced Pattern Matching + Clean Case (Elixir Notes)

---

## 🧠 New Concept: Atoms

### What is this?

```elixir
:ok
:error
```

These are **atoms**.

---

## 🔹 Think of Atoms Like:

> Strings for internal use (not for users)

---

## 🔸 Difference: Atom vs String

| Type   | Purpose                        | Example            |
| ------ | ------------------------------ | ------------------ |
| String | User-facing text               | `"File not found"` |
| Atom   | Internal status / control flow | `:ok`, `:error`    |

---

## 🧠 Rule of Thumb

* Use **strings** → user messages
* Use **atoms** → system signals

---

## 🔹 Common Atoms

```elixir
:ok
:error
```

Used everywhere in Elixir

---

## 🔥 Important Concept (HARD PART)

### Pattern Matching can:

> ✅ Assign values
> ✅ Compare values
> ❗ Do BOTH at the same time

---

## 🔹 Example 1 (Basic)

```elixir
color = "red"
```

* Assigns `"red"` to `color`

---

## 🔹 Example 2 (List Matching)

```elixir
[color] = ["red"]
```

* Extracts value from list

---

## 🔥 Example 3 (Hardcoded Matching)

```elixir
["red", color] = ["red", "blue"]
```

✅ Works:

* First element matches `"red"`
* Second assigned → `color = "blue"`

---

## ❌ Example 4 (Mismatch)

```elixir
["green", color] = ["red", "blue"]
```

❌ Fails:

* `"green"` ≠ `"red"`
* Pattern does NOT match → crash

---

## 🧠 Key Insight

> If you hardcode a value on the left, it MUST match the right

---

## 🔹 Applying This to `case`

### Old Version

```elixir
{status, binary} = File.read(file_name)

case status do
  :ok ->
    :erlang.binary_to_term(binary)

  :error ->
    "That file does not exist"
end
```

---

## 🔥 Improved Version (Pattern Matching Inside Case)

```elixir
def load(file_name) do
  case File.read(file_name) do
    {:ok, binary} ->
      :erlang.binary_to_term(binary)

    {:error, reason} ->
      "That file does not exist"
  end
end
```

---

## 🔍 What’s Happening

### Case 1:

```elixir
{:ok, binary}
```

* Matches ONLY if:

  * First element = `:ok`
* Assigns second element → `binary`

---

### Case 2:

```elixir
{:error, reason}
```

* Matches ONLY if:

  * First element = `:error`
* Assigns second element → `reason`

---

## 🔥 Powerful Idea

> Matching + Assigning in ONE step

---

## 🧠 Mental Model

```elixir
case value do
  pattern -> result
end
```

* Elixir tries patterns **top to bottom**
* First match wins

---

## ⚠️ Warning: Unused Variable

```elixir
{:error, reason}
```

If `reason` not used → ⚠️ warning

---

## ✅ Fix: Ignore Variable

```elixir
{:error, _reason}
```

* `_` → “I don’t care about this value”

---

## 🔹 Final Clean Code

```elixir
def load(file_name) do
  case File.read(file_name) do
    {:ok, binary} ->
      :erlang.binary_to_term(binary)

    {:error, _reason} ->
      "That file does not exist"
  end
end
```

---

## 🔥 Why This is Powerful

* No `if` statements
* No manual checks
* Structure-driven logic
* Clean + safe

## 🆚 JavaScript vs Elixir

### JavaScript

```javascript
const result = readFile(fileName);

if (result.status === "ok") {
  return decode(result.data);
} else {
  return "File not found";
}
```

---

### Elixir

```elixir
case File.read(file_name) do
  {:ok, binary} -> ...
  {:error, _} -> ...
end
```

---

## 🧠 Key Takeaways

* Atoms = internal signals (`:ok`, `:error`)
* Pattern matching can compare + assign
* Hardcoded values must match exactly
* Use `_variable` to ignore values
* `case` + pattern matching = idiomatic Elixir

---

## 🚀 Big Insight

> Elixir doesn’t “check conditions”
> It **matches patterns**

---

👉 This is one of the most important mental shifts in Elixir.
